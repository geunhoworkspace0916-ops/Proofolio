# 이더리움 세폴리아 기반 검증 가능한 디지털 증명서 플랫폼 (Proofolio) PRD/SRS 및 화면 설계서

문서 버전: 1.0
작성일: 2026-05-26
대상 네트워크: Ethereum Sepolia Testnet
대상 프로젝트: 수업 기말 프로젝트 겸 포트폴리오(이력서)용 블록체인 dApp
문서 목적: 검증된 발급기관이 경력·교육 수료·프로젝트·수상 증명서를 온체인으로 발급하고, 검증자가 QR/링크로 진위를 확인하는 서비스(Proofolio)의 PRD, SRS, 화면 설계서, 구현계획을 하나의 문서로 정의한다. 본 문서는 "동작하는 토이"가 아니라 **실제 서비스로 내놔도 손색없는 제품**을 만들기 위한 기준선이다.

## 1. 개요

### 1.1 제품 개요

Proofolio는 학교·부트캠프·회사·교수·대회 운영진 같은 **검증된 발급기관(Verified Issuer)**이 학생/지원자에게 **위변조 불가능한 디지털 증명서**를 발급하고, 채용 담당자·심사위원 같은 **검증자(Verifier)**가 QR 또는 링크로 **진위를 즉시 확인**할 수 있는 블록체인 기반 증명서 플랫폼이다.

핵심은 "증명서 PDF 자체를 믿는 것"이 아니라 **발급기관의 온체인 발급 기록과 서명을 믿는 것**이다. 증명서의 원본 파일은 오프체인에 두고, 온체인에는 발급자 주소, 보유자 주소, 파일 해시, 발급 시각, 취소 여부만 기록한다. 사용자는 자기 지갑에 양도 불가능한 증명서(소울바운드 토큰)를 받고, 이력서에 검증 링크/QR을 넣는다. 검증자는 링크를 열어 "등록된 기관이 발급했는가", "취소되지 않았는가", "원본 파일 해시가 일치하는가"를 확인한다.

### 1.2 핵심 설계 결정

| 항목 | 설계 결정 |
|---|---|
| 신뢰 모델 | 증명서 파일이 아니라 발급기관의 온체인 발급 기록을 신뢰 |
| 발급기관 정의 | 정부 "공인"이 아니라 레지스트리에 등록된 "검증된 발급기관(Verified Issuer)" |
| 발급기관 등록 | 플랫폼 관리자(Admin)가 발급기관을 레지스트리에 등록/비활성 |
| 증명서 형태 | 보유자 지갑에 발급되는 **소울바운드(양도 불가) ERC-721** |
| 온체인 저장 | 발급자·보유자·파일 해시·종류·발급시각·취소여부만 저장(원본 파일은 오프체인) |
| 무결성 검증 | 원본 파일을 검증자가 업로드 → 해시 계산 → 온체인 해시와 비교 |
| 취소 | 발급한 기관(또는 관리자)이 증명서를 취소(Revocation) 가능 |
| 검증 접근 | 검증은 지갑 없이 누구나 가능(읽기 전용) |
| 컨트랙트 구조 | 단일 `Proofolio` 컨트랙트(발급기관 레지스트리 + 증명서 SBT) |
| 규모 가정 | 동시 접속 소수, 발급기관 수십, 증명서 수백 규모(학습/시연) |

### 1.3 용어 정의

| 용어 | 정의 |
|---|---|
| 발급기관(Issuer) | 레지스트리에 등록되어 증명서를 발급할 수 있는 주체(학교·부트캠프·회사·교수·대회) |
| 사용자/보유자(Holder) | 증명서를 자기 지갑에 받는 학생·수강생·지원자 |
| 검증자(Verifier) | QR/링크로 증명서 진위를 확인하는 채용 담당자·심사위원 |
| 관리자(Admin) | 발급기관을 레지스트리에 등록/비활성하는 플랫폼 운영 주체(컨트랙트 owner) |
| 증명서(Credential) | 발급자·보유자·해시·종류·시각을 담아 발급되는 소울바운드 토큰 |
| 데이터 해시(dataHash) | 증명서 원본 파일의 keccak256 해시. 위변조 검증 기준 |
| 소울바운드(SBT) | 양도 불가능한 토큰. 증명서가 거래되지 않도록 전송을 막음 |
| 취소(Revocation) | 발급된 증명서를 무효화하는 행위. 기록은 남고 상태가 "취소됨"이 됨 |
| 발급기관 프로필 | 발급기관의 이름·소개를 담은 공개 페이지 |

### 1.4 신뢰 모델 판단 (블록체인이 보장하는 것과 아닌 것)

본 프로젝트의 가치와 한계를 명확히 하기 위해 신뢰 모델을 다음과 같이 정의한다.

블록체인이 **보장하는 것**:

- 이 증명서를 **등록된 발급기관이** 발급했다.
- **언제** 발급했는지(발급 시각).
- 발급 기록을 나중에 **몰래 수정할 수 없다**.
- 증명서 원본 파일이 바뀌면 **해시가 불일치**한다.
- 서비스 운영자를 완전히 믿지 않아도 **누구나 검증**할 수 있다.

블록체인이 **직접 보장하지 않는 것**:

- 그 사람이 실제로 얼마나 실력이 좋은가.
- 그 발급기관이 정말 좋은/정당한 기관인가.
- 발급자가 실수로/부정하게 발급하지 않았는가.
- 온체인에 기록되는 `dataHash`는 **발급기관이 제출한 값을 그대로 신뢰**한다(발급 시점의 발급기관 정직성을 가정). 컨트랙트가 파일을 직접 해싱하지는 않는다.
- 발급기관 **이름은 자기신고 문자열**이라 사칭/중복이 가능하다. 따라서 신뢰의 기준은 *이름*이 아니라 **발급기관 주소 + 관리자 레지스트리 등록 사실**이며, 검증 UI는 주소를 항상 함께 노출한다.

추가 기술적 한계 — **파일 무결성 대조는 "정확히 같은 바이트"일 때만 성립**한다. 같은 내용이라도 PDF를 재저장하거나 재인코딩하면 바이트가 달라져 해시가 불일치(오탐)할 수 있다. 따라서 "발급 당시의 원본 파일 그대로"를 검증 대상으로 한다(검증 화면에 안내).

따라서 **발급기관 등록(레지스트리), 취소(Revocation), 발급기관 공개 프로필**이 신뢰를 보완하는 핵심 장치다. 이 한계를 발표/면접에서 솔직히 설명하면 "블록체인을 언제 써야/말아야 하는지 안다"는 성숙도를 보일 수 있다.

### 1.5 포함 범위

- 웹 브라우저 기반 프론트엔드, MetaMask 지갑 연결, Sepolia 네트워크 확인/전환 안내
- 관리자: 발급기관 등록 및 활성/비활성 관리
- 발급기관: 증명서 발급(보유자 주소·파일 해시·종류·메타데이터), 증명서 취소
- 사용자: 내 증명서 목록 조회
- 검증자: QR/링크 기반 증명서 검증, 원본 파일 업로드 해시 비교
- 발급기관 공개 프로필 페이지
- 소울바운드 ERC-721 기반 증명서, Etherscan 트랜잭션 링크
- Sepolia 컨트랙트 배포(가능 시 verify) + Cloudflare Pages 웹 배포 + Hardhat 테스트
- 실서비스급 디자인/UX(다크모드·반응형·트랜잭션 토스트·랜딩)

### 1.6 제외 범위

- 발급기관의 실재성/정당성 자동 심사(현실의 DID/트러스트 레지스트리 영역)
- 증명서 원본 파일의 온체인 저장(해시만 저장)
- 증명서 내용의 영구 비공개(ZK 기반 선택적 공개) — 본 범위 밖
- 유료 결제/구독, ERC-20 연동
- 대규모 인덱싱 백엔드(온체인 조회 + 경량 프론트로 대체)

### 1.7 주요 가정

- 발급기관·검증자·사용자는 약간의 Sepolia 테스트 ETH를 보유하거나(발급/취소 트랜잭션), 검증은 지갑 없이 읽기 전용으로 수행한다.
- 증명서 원본 파일(PDF/이미지)은 발급기관/사용자가 별도 보관하며, 검증 시 검증자에게 제공된다.
- 동시 접속자는 소수, 발급기관 수십, 증명서 수백 규모다.
- 관리자는 신뢰 가능한 단일 운영 주체(컨트랙트 owner)로 가정한다.

## 2. PRD

### 2.1 문제 정의

현재 이력서·지원서에는 다음 위변조 문제가 있다.

- 가짜 수료증 PDF, 조작된 프로젝트 참여 이력
- 실제로 다니지 않은 부트캠프/교육 이력, 회사 경력 과장
- 수상 내역 위조, 이미지/PDF 편집으로 만든 인증서

기존 검증 방식은 채용 담당자가 발급기관에 일일이 연락하거나, 제출된 PDF를 그대로 믿어야 한다. 데이터베이스 기반 시스템은 운영자가 나중에 기록을 바꿀 수 있어 "운영자 신뢰"가 전제된다. Proofolio는 발급 기록을 온체인에 남겨 **운영자조차 수정할 수 없고 누구나 검증 가능한** 구조로 이 문제를 해결한다.

### 2.2 제품 목표

| 구분 | 목표 |
|---|---|
| 핵심 가치 | 운영자를 신뢰하지 않아도 검증 가능한 위변조 불가 증명서 발급/검증 |
| 사용자 가치 | 사용자는 평생 소유 가능한 검증 가능 증명서, 검증자는 즉시 진위 확인 |
| 기술 목표 | 발급기관 레지스트리·소울바운드 발급·취소·해시 검증을 정확히 구현하고 테스트로 검증 |
| 포트폴리오 목표 | 면접에서 깊이 설명 가능한 이해 + 실서비스급 완성도/디자인 |
| 비목표 | 발급기관 정당성 자동 심사, 내용 진위 보증, 대규모 트래픽 처리 |

### 2.3 사용자 유형

| 역할 | 설명 |
|---|---|
| 관리자(Admin) | 발급기관을 레지스트리에 등록/비활성. 컨트랙트 owner. |
| 발급기관(Issuer) | 등록된 활성 기관. 증명서를 발급/취소한다. |
| 사용자/보유자(Holder) | 증명서를 자기 지갑에 받는 학생·지원자. |
| 검증자(Verifier) | QR/링크로 증명서를 확인하는 채용 담당자·심사위원. 지갑 없이 가능. |

### 2.4 핵심 사용자 시나리오

#### 관리자 시나리오
1. 지갑을 연결하고 Sepolia 네트워크임을 확인한다.
2. 발급기관 관리 화면에서 기관 주소·이름·소개를 입력해 발급기관으로 등록한다.
3. 필요 시 특정 발급기관을 비활성화한다.

#### 발급기관 시나리오
1. 지갑을 연결한다(등록된 활성 발급기관).
2. 증명서 발급 화면에서 보유자 주소, 증명서 종류(수료/경력/프로젝트/수상), 원본 파일을 업로드한다. dApp이 파일 해시를 계산한다.
3. 발급을 실행하면 보유자 지갑에 소울바운드 증명서가 발급되고 발급 기록이 온체인에 남는다.
4. 잘못 발급한 경우 해당 증명서를 취소한다.

#### 사용자(보유자) 시나리오
1. 지갑을 연결해 "내 증명서 목록"에서 받은 증명서를 확인한다.
2. 이력서/포트폴리오에 증명서 검증 링크/QR을 넣는다.

#### 검증자 시나리오
1. 이력서의 QR/링크를 연다(지갑 불필요).
2. "등록된 기관이 발급했는가", "취소되지 않았는가", "발급 시각", "발급기관 활성 여부"를 확인한다.
3. (선택) 지원자가 제출한 원본 파일을 업로드해 온체인 해시와 일치하는지 대조한다.

### 2.5 제품 요구사항

- PR-1: 등록된 활성 발급기관만 증명서를 발급할 수 있어야 한다.
- PR-2: 발급 기록(발급자·보유자·해시·시각)은 변조 불가하게 온체인에 남아야 한다.
- PR-3: 검증자는 지갑 없이도 QR/링크로 진위를 확인할 수 있어야 한다.
- PR-4: 원본 파일 위변조는 해시 비교로 탐지할 수 있어야 한다.
- PR-5: 증명서는 양도 불가능해야 한다(소울바운드).
- PR-6: 잘못 발급된 증명서는 취소할 수 있어야 한다.
- PR-7: UI/UX는 실서비스 수준의 완성도와 신뢰감을 제공해야 한다.

### 2.6 성공 지표

| 지표 | 목표 |
|---|---|
| 기능 완결성 | 발급기관 등록→발급→검증→취소 전 흐름이 Sepolia에서 무오류 동작 |
| 검증 정확성 | 정상 증명서 ✅, 변조 파일 ❌, 취소 증명서 "취소됨"이 정확히 표시 |
| 데모 성공률 | 5~7분 라이브 데모를 대기 없이 완주(백업 영상 보유) |
| 적절성 설득력 | "왜 블록체인" 질문에 신뢰 모델로 즉답 가능 |
| 완성도 인상 | 처음 접속한 사람이 "실제 서비스 같다"고 느끼는 디자인/UX |

### 2.7 제약사항

- 블록체인의 모든 공개 데이터는 영구·공개다. 따라서 개인 식별 정보는 온체인에 직접 저장하지 않고 해시/메타데이터만 다룬다.
- 발급기관의 실재성/정당성은 본 시스템이 자동 보증하지 않는다(관리자 등록 + 공개 프로필로 보완).
- 가스 비용·블록 확정 지연으로 발급/취소에 수 초~수십 초 지연이 있을 수 있다.
- 원본 파일은 오프체인 보관이므로 분실 시 무결성 대조(해시 비교)는 불가하다(발급 사실/취소 여부는 여전히 검증 가능).

## 3. SRS

### 3.1 시스템 구성

```text
[브라우저: React dApp]
   │  ethers.js v6 (provider/signer)
   │  파일 → keccak256 해시 (클라이언트 계산)
   ▼
[MetaMask 지갑] ──▶ [Ethereum Sepolia]
                         │
                         ▼
                 [Proofolio 컨트랙트]
                 - 발급기관 레지스트리(Admin 관리)
                 - 증명서 발급(소울바운드 ERC-721)
                 - 취소(Revocation), 검증 view
                 - 이벤트 발행 → 프론트가 조회/구독

[검증자] ── QR/링크(verify/{tokenId}) ──▶ 읽기 전용 조회(지갑 불필요)
[오프체인] ── 증명서 원본 파일(PDF/이미지) 보관, 검증 시 해시 대조
```

- 검증은 읽기 전용 RPC로 지갑 없이 가능하다.
- 파일 해시는 클라이언트에서 계산하며, 원본 파일은 온체인에 올리지 않는다.

### 3.2 권장 기술 스택

| 구분 | 기술 |

| 스마트컨트랙트 | Solidity ^0.8.x, OpenZeppelin(ERC721, Ownable) |
| 개발/테스트 | Hardhat, hardhat-toolbox, chai, ethers |
| 프론트엔드 | React 19, TypeScript, Vite |
| 디자인/UI | Tailwind CSS, Radix UI 패턴(shadcn/ui 스타일), lucide 아이콘, 디자인 토큰, 다크/라이트 |
| 블록체인 연동 | ethers.js v6, MetaMask(EIP-1193) |
| 보조 | QR **생성**(필수, qrcode) — 스캔은 휴대폰 카메라가 링크를 열므로 앱 내 스캔은 선택. 파일 해시: `ethers.keccak256(new Uint8Array(await file.arrayBuffer()))` |
| 네트워크 | Ethereum Sepolia |
| 배포 | 컨트랙트: Hardhat + Sepolia RPC / 웹: Cloudflare Pages(wrangler) |

### 3.3 발급/검증 구조

- 단일 `Proofolio` 컨트랙트가 **발급기관 레지스트리 + 증명서 SBT**를 함께 관리한다.
- 발급기관 등록은 관리자(owner)만, 증명서 발급/취소는 등록된 활성 발급기관만 수행한다.
- 증명서는 보유자 주소로 민팅되는 소울바운드 ERC-721이며, 토큰별로 발급 메타데이터를 저장한다.
- 검증은 `tokenId` 기준 view 함수로 누구나 조회한다. QR/링크는 `verify/{tokenId}` 경로를 가리킨다.

### 3.4 증명서 / 발급기관 상태 정의

| 대상 | 값 | 상태 |
|---|---|---|
| 발급기관 | Active | 발급 가능한 등록 기관 |
| 발급기관 | Inactive | 비활성(신규 발급 불가, 기존 발급 기록은 유지) |
| 증명서 | Valid | 유효한 증명서 |
| 증명서 | Revoked | 취소된 증명서(기록은 남고 "취소됨" 표시) |

### 3.6 기능 요구사항

#### FR-001. 지갑 연결
- 사용자는 화면에서 지갑 연결 버튼을 클릭할 수 있어야 한다.
- 시스템은 MetaMask 설치 여부를 확인해야 한다.
- 연결 성공 시 축약 주소를 표시하고, 연결 주소가 관리자/등록 발급기관이면 해당 배지를 표시해야 한다.
- 연결 실패/거절 시 오류 메시지를 표시해야 한다.

#### FR-002. Sepolia 네트워크 확인
- 시스템은 연결된 지갑의 체인 ID가 Sepolia인지 확인해야 한다.
- Sepolia가 아니면 발급기관 등록·발급·취소 버튼을 비활성화해야 한다.
- 네트워크 전환 버튼 또는 안내를 제공해야 한다.

#### FR-003. 발급기관 등록 (관리자)
- 발급기관 등록은 관리자(owner)만 수행할 수 있어야 한다.
- 입력값은 발급기관 주소, 이름, 소개/프로필 URI여야 한다.
- 주소·이름은 필수값이어야 한다.
- 이미 등록된 주소를 다시 등록하면 거부해야 한다.
- 등록 성공 후 발급기관 목록에 반영되어야 한다.

#### FR-004. 발급기관 활성/비활성 관리 (관리자)
- 관리자는 등록된 발급기관을 비활성/재활성할 수 있어야 한다.
- 비활성 기관은 신규 발급을 할 수 없어야 한다.
- 비활성화는 기존 발급 기록의 발급 사실을 삭제하지 않으며, 검증 화면에 "발급기관 현재 비활성" 표시를 제공해야 한다.

#### FR-005. 발급기관 목록/프로필 조회
- 모든 사용자는 등록된 발급기관 목록과 공개 프로필을 조회할 수 있어야 한다.
- 각 발급기관은 이름, 주소(축약), 활성 여부, 발급 수를 표시해야 한다.

#### FR-006. 증명서 발급 (발급기관)
- 증명서 발급은 등록된 활성 발급기관만 수행할 수 있어야 한다.
- 입력값은 보유자 주소, 증명서 종류(수료/경력/프로젝트/수상 등), 원본 파일(해시 계산용), 선택적 메타데이터 URI여야 한다.
- 발급 성공 시 보유자 지갑으로 소울바운드 증명서가 민팅되어야 한다.
- 발급 기록(발급자·보유자·해시·종류·시각)이 온체인에 저장되어야 한다.

#### FR-007. 증명서 파일 해시 생성/저장
- 발급 시 dApp은 원본 파일의 keccak256 해시를 클라이언트에서 계산해야 한다.
- 원본 파일은 온체인에 저장하지 않고 해시만 저장해야 한다.

#### FR-008. 내 증명서 목록 조회 (사용자)
- 사용자는 자기 지갑이 보유한 증명서 목록을 조회할 수 있어야 한다.
- 각 증명서는 발급기관, 종류, 발급일, 유효/취소 상태, 검증 링크/QR을 표시해야 한다.

#### FR-009. 증명서 검증 (검증자)
- 검증자는 QR/링크(`verify/{tokenId}`)로 증명서를 조회할 수 있어야 한다(지갑 불필요).
- 검증 결과는 발급기관(이름/주소/활성여부), 보유자, 발급일, 유효/취소 상태를 표시해야 한다.

#### FR-010. 파일 원본 해시 대조
- 검증자는 지원자가 제출한 원본 파일을 업로드할 수 있어야 한다.
- 시스템은 업로드 파일의 keccak256 해시를 계산해 온체인 dataHash와 비교해야 한다.
- 일치하면 "원본 일치", 불일치하면 "위변조 의심"을 명확히 표시해야 한다.

#### FR-011. 증명서 취소 (발급기관)
- 증명서를 **발급한 기관**이 해당 증명서를 취소할 수 있어야 한다(관리자는 직접 취소하지 않고, 필요 시 발급기관을 비활성화한다).
- 취소된 증명서는 검증 시 "취소됨"으로 표시되어야 한다.
- 취소는 발급 기록을 삭제하지 않으며 상태만 변경해야 한다.

#### FR-012. 트랜잭션 상태 표시
- 시스템은 서명 대기, 전송 중, 확정 대기, 성공, 실패 상태를 구분 표시해야 한다.
- 트랜잭션 해시 생성 시 Sepolia Etherscan 링크를 제공해야 한다.

#### FR-013. 발급기관 공개 프로필
- 각 발급기관은 이름·소개·발급 이력 요약을 담은 공개 프로필 페이지를 가져야 한다.
- 검증 화면에서 발급기관 프로필로 이동할 수 있어야 한다.

#### FR-014. 테마 전환
- 사용자는 다크/라이트 모드를 토글할 수 있어야 하며, 설정은 유지되어야 한다.

### 3.7 스마트 컨트랙트 요구사항

#### SC-001. 발급기관 레지스트리
- 컨트랙트는 발급기관을 `registerIssuer(address, name, metaURI)`로 등록해야 한다(관리자 전용).
- 발급기관별 이름, 프로필 URI, 활성 여부, 등록 시각을 저장해야 한다.
- 이미 등록된 주소 재등록은 거부해야 한다.

#### SC-002. 발급기관 권한/활성 상태
- `setIssuerActive(address, bool)`로 활성/비활성을 전환할 수 있어야 한다(관리자 전용).
- 발급 함수는 `활성 발급기관`만 호출할 수 있어야 한다.

#### SC-003. 증명서 발급 (소울바운드 ERC-721)
- `issueCredential(holder, dataHash, credType, metaURI)`는 등록된 활성 발급기관만 호출할 수 있어야 한다.
- 입력 검증: `holder != address(0)`, `dataHash != bytes32(0)`, `credType` 비어있지 않음(아니면 revert).
- 호출 시 새 `tokenId`를 발급하고 `holder`에게 민팅해야 한다. **`_safeMint`가 아니라 `_mint`를 사용**한다(임의의 holder가 컨트랙트일 때 `onERC721Received` 콜백을 통한 재진입/그리핑을 피하기 위함).
- 상태 변경(저장) → 이벤트 순서로 처리(checks-effects-interactions).
- `CredentialIssued` 이벤트를 발행해야 한다.

#### SC-004. 증명서 데이터/해시 저장
- 토큰별로 발급자, 보유자, dataHash(keccak256), 종류, 메타데이터 URI, 발급 시각, 취소 여부를 저장해야 한다.
- 원본 파일은 저장하지 않는다.

#### SC-005. 증명서 취소(Revocation)
- `revokeCredential(tokenId)`는 **그 증명서를 발급한 기관만** 호출할 수 있어야 한다. (관리자는 임의 증명서를 직접 취소하지 못하며, 문제가 있으면 `setIssuerActive`로 발급기관을 비활성화한다 → 중앙 권한 남용 방지.)
- 존재하지 않는 `tokenId` 취소는 거부해야 한다.
- 취소 시 `revoked = true`로 설정하고 `CredentialRevoked` 이벤트를 발행해야 한다.
- 이미 취소된 증명서 재취소는 거부해야 한다.
- 취소는 토큰을 소각하지 않고 상태 플래그만 변경한다(기록 보존·감사 가능).

#### SC-006. 소울바운드(양도 불가)
- `_update(to, tokenId, auth)`를 오버라이드해 **발급(mint, `from == address(0)`)만 허용**하고, 보유자 간 전송(`from != 0 && to != 0`)은 revert해야 한다.
- ⚠️ 구현 주의: mint 경로까지 막으면 발급 자체가 불가능해지는 흔한 실수가 있으므로, `from == address(0)`은 반드시 통과시켜야 한다.
- 본 설계는 취소를 소각이 아닌 플래그로 처리하므로 burn 경로는 사용하지 않는다(원하면 발급기관 burn을 별도 허용 가능).

#### SC-007. 검증 view 함수
- `verify(tokenId)`는 발급자, 발급자명, 발급자 활성 여부, 보유자, dataHash, 종류, 발급 시각, 취소 여부를 반환해야 한다.
- `credentialsOf(holder)`는 보유자의 토큰 목록을 반환해야 한다.
  - 구현 시 `mapping(address => uint256[])`를 발급(mint) 시점에 갱신해 두고 그 배열을 반환한다. **상태 변경 함수 안에서는 전체 토큰을 순회하는 무제한 루프를 두지 않는다**(가스 폭증·DoS 방지). 대량 목록은 프론트엔드가 `CredentialIssued` 이벤트 인덱싱으로 보완한다.
- `isValid(tokenId)`는 (존재 && !취소 && 발급자 활성)을 반환해야 한다.

#### SC-008. 이벤트

| 이벤트 | 발생 시점 | 주요 데이터 |
|---|---|---|
| IssuerRegistered | 발급기관 등록 | issuer, name |
| IssuerStatusChanged | 활성/비활성 변경 | issuer, active |
| CredentialIssued | 증명서 발급 | tokenId, issuer, holder, dataHash, credType |
| CredentialRevoked | 증명서 취소 | tokenId, issuer |

#### SC-009. 접근 제어
- 관리자 함수(`registerIssuer`, `setIssuerActive`)는 owner만 호출. owner 이전은 **`Ownable2Step`**(2단계 수락)으로 안전하게 처리하고, 키 분실/오전송 위험을 줄인다.
- 발급은 `활성 발급기관`, 취소는 `발급한 기관` modifier로 강제.
- 모든 입력값 유효성 검증: 주소 `!= address(0)`, 해시 `!= 0`, 이름/종류 비어있지 않음, 중복 등록·중복 취소 차단.
- 발급기관 키 탈취 대응: 관리자가 해당 기관을 즉시 비활성화하고, 영향받은 증명서는 해당 기관이 취소한다.

### 3.9 비기능 요구사항

#### NFR-001. 접근성
- 명도 대비 WCAG AA 지향, 키보드 내비게이션, 포커스 링, 이미지 alt, 버튼 비활성 사유 표시.

#### NFR-002. 성능
- 검증은 읽기 전용 RPC로 지갑 없이 빠르게 동작. 목록 조회 캐싱/스켈레톤. 첫 로드 체감 2초 이내.

#### NFR-003. 보안
- 접근 제어(owner/issuer), 입력 검증, 소울바운드 전송 차단, 정수 오버플로(^0.8 기본).
- 개인정보 온체인 직접 저장 금지(해시/메타만).

#### NFR-004. 신뢰성/검증가능성
- 모든 발급/취소가 이벤트·view로 재구성 가능. Etherscan 독립 검증(verify 권장).
- 검증 결과는 온체인 데이터와 항상 일치.

#### NFR-005. 사용성
- 검증 결과를 ✅/⚠️/❌로 직관 표시, QR/링크 공유 용이, 트랜잭션 토스트, 빈/에러/로딩 상태, 반응형, 다크/라이트.

#### NFR-006. 이식성/유지보수
- 컨트랙트 주소·RPC를 환경변수로 주입. 타입 안전(TS). voting-dapp 패턴 재사용.

#### NFR-007. 프라이버시
- 온체인 데이터는 **영구적이고 누구나 열람 가능**하다. 따라서 `credType`·`metaURI`에 이름·이메일·학번 같은 **개인식별정보(PII)를 절대 넣지 않는다**(해시·일반 종류명만).
- `credentialsOf(holder)`와 이벤트로 특정 주소가 보유한 모든 증명서가 **공개·열거 가능**하다. 지갑 주소가 실명과 연결되면 신원이 드러날 수 있음을 사용자에게 고지한다.
- 권장: 보유자는 용도별로 분리된 지갑 사용 가능. 온체인 저장은 최소화하고 상세는 오프체인에 둔다.
- (한계) 완전한 프라이버시(선택적 공개)는 ZK가 필요하며 본 범위 밖이다.

### 4.2.1 화면 흐름도 (역할별 진입 및 이동)

역할에 따라 들어가는 화면이 다르다. **검증(UI-005)은 지갑 없이 누구나** 들어갈 수 있다.

```text
                          [UI-000 홈 / 랜딩]
                                  │
      ┌──────────────┬────────────┴───────────┬───────────────┐

 "검증하기"     "발급(기관)"              "내 증명서"     "기관 관리(관리자)"
      │              │                         │               │
      │              ▼                         ▼               ▼
      │       [UI-001 지갑 연결]        [UI-001 지갑 연결] [UI-001 지갑 연결]
      │              │                         │               │
      │              ▼                         ▼               ▼
      │       [UI-003 증명서 발급]      [UI-004 내 증명서] [UI-002 기관 관리]
```

### 4.6 UI-003. 증명서 발급 (발급기관)

#### 기능 명세
| 항목 | 내용 |
|---|---|
| 목적 | 보유자에게 증명서 발급 |
| 접근 권한 | 등록된 활성 발급기관 |
| 입력값 | 보유자 주소, 증명서 종류, 원본 파일(해시), 메타 URI |

1. 발급기관이 **원본 파일을 선택** → ② 브라우저가 그 파일의 **지문(keccak256 해시)을 계산**(파일은 업로드되지 않음, 내 PC에서 계산) → ③ **[증명서 발급]** 누르면 보유자 지갑으로 소울바운드 증명서가 민팅되고 (발급자·보유자·해시·종류·시각)이 온체인에 기록됨.

```text
+------------------------------------------------------------------------+

|  증명서 발급                              발급기관: 코드스쿨 부트캠프    |
|------------------------------------------------------------------------|

|  증명서 종류   [ 수료증 ▾ ]   (수료 / 경력 / 프로젝트 / 수상)           |

|  원본 파일     [ 📎 certificate.pdf 선택됨 ]                            |

|        ⓘ 파일은 업로드되지 않아요. 브라우저에서 해시만 계산합니다.       |
```

#### 구성 요소 및 동작
| 요소 | 설명 / 동작 |
|---|---|
| 보유자 지갑 주소 | 증명서를 받을 사람의 주소(필수). 이 주소로 토큰이 민팅됨. |
| 증명서 종류 | 드롭다운(수료/경력/프로젝트/수상). 검증 화면에 표시됨. |
| 원본 파일 선택 | 파일 선택 시 브라우저가 **keccak256 해시**를 즉시 계산(파일은 서버/체인에 안 올라감). |
| 파일 지문(해시) | 계산된 해시 미리보기. 이 값이 온체인에 저장되어 나중에 위변조 대조 기준이 됨. |
| 메타 URI | 선택. 추가 메타데이터(IPFS/URL) 링크. |
| **[증명서 발급]** | `issueCredential(보유자, 해시, 종류, 메타)` 트랜잭션 호출. 미등록/비활성 기관이면 `ERR_NOT_ISSUER`. 성공 시 토큰 ID 발급 + 완료 화면. |
| 완료 화면 버튼 | 검증 링크 복사 / QR 보기 / Etherscan 확인 / 이어서 발급. |

#### 상태
- **권한 없음**: 등록 활성 발급기관이 아니면 폼 대신 안내(관리자에게 등록 요청 문구).
- **검증 실패**: 보유자 주소 형식 오류·파일 미선택·종류 미선택 시 해당 필드에 빨간 안내(트랜잭션 전 차단).
- **트랜잭션**: 발급 중 UI-007 토스트(서명 대기→전송→확정). 거절 시 `ERR_TX_REJECTED`.

### 4.8 UI-005. 증명서 검증 (검증자)

#### 기능 명세
| 항목 | 내용 |
|---|---|
| 목적 | QR/링크로 진위 확인 + 원본 파일 해시 대조 |
| 접근 권한 | 누구나(지갑 불필요) |
| 입력 | 증명서 ID/링크, (선택) 원본 파일 |
| 표시 | 발급기관(이름/활성), 보유자, 발급일, 유효/취소, 해시 일치 여부 |

진입 & 동작 (지갑 불필요):
1. 검증자가 이력서의 **QR을 스캔**하거나 **링크(`verify/{id}`)를 클릭**, 또는 홈에서 **증명서 ID를 입력**해 들어온다.
2. 화면이 `verify(tokenId)`를 **읽기 전용으로 조회**(트랜잭션·지갑 불필요)해 발급기관·발급일·취소여부를 표시한다.
3. (선택) 지원자가 제출한 **원본 파일을 업로드**하면, 브라우저가 그 파일의 keccak256 해시를 계산해 **온체인에 기록된 해시와 비교** → 같으면 ✅ 원본 일치, 다르면 ❌ 위변조 의심.

#### 상태 1 — 유효 + 원본 일치 (정상)
```text
+------------------------------------------------------------------------+
|  증명서 검증  #12                                          (지갑 불필요)|
+------------------------------------------------------------------------+
|   ╭────────────────────────────────────────────────────────────────╮  |
|   │  ✅  유효한 증명서                                               │  |
|   ╰────────────────────────────────────────────────────────────────╯  |
|   종류          수료증                                                 |
|   발급기관      코드스쿨 부트캠프   🟢 활성   [ 기관 프로필 ▸ ]        |
|   발급기관 주소  0xABCD...1234                                         |
|   보유자        0xHODL...beef                                          |
|   발급일        2026-03-01                                            |
|------------------------------------------------------------------------|
|   원본 파일 대조(선택)  [ 📎 파일 업로드 ]                              |
|   → ✅ 원본 일치   (해시 0x7a3f9c...e2 = 온체인 기록과 동일)            |
|                                              [ Etherscan에서 보기 ↗ ]  |
+------------------------------------------------------------------------+
```

#### 상태 2 — 위변조 의심 (해시 불일치)
```text
+------------------------------------------------------------------------+
|  증명서 검증  #12                                                      |
|   ✅ 유효한 증명서 (발급 기록은 진짜)                                   |
|   발급기관  코드스쿨 부트캠프 🟢활성   발급일 2026-03-01               |
|------------------------------------------------------------------------|
|   원본 파일 대조  [ 📎 수정된파일.pdf 업로드됨 ]                        |
|   → ❌ 해시 불일치 — 제출된 파일이 발급 당시 원본과 다릅니다(위변조 의심)|
|      온체인 해시 0x7a3f...e2  ≠  업로드 해시 0x91bd...4c               |
+------------------------------------------------------------------------+
```

#### 상태 3 — 취소된 증명서
```text
+------------------------------------------------------------------------+
|  증명서 검증  #34                                                      |
|   ╭────────────────────────────────────────────────────────────────╮  |
|   │  ❌  취소된 증명서 (Revoked)                                     │  |
|   ╰────────────────────────────────────────────────────────────────╯  |
|   발급기관  코드스쿨 부트캠프   발급일 2026-02-10   취소됨            |
|   ⓘ 발급 기록은 남아 있으나 발급기관이 이 증명서를 취소했습니다.        |
+------------------------------------------------------------------------+
```

#### 상태 4 — 발급기관 비활성 (주의)
```text
+------------------------------------------------------------------------+
|  증명서 검증  #20                                                      |
|   ⚠️  발급 기록은 유효하나 발급기관이 현재 비활성 상태입니다           |
|   발급기관  XYZ 교육원  ⚪비활성   발급일 2025-12-01                   |
|   ⓘ "발급 사실"은 변하지 않지만, 기관 신뢰도는 별도로 판단하세요.       |
+------------------------------------------------------------------------+
```

#### 상태 5 — 존재하지 않는 증명서
```text
+------------------------------------------------------------------------+
|  증명서 검증                                                           |
|   🔍 해당 ID(#999)의 증명서를 찾을 수 없습니다. ID/링크를 확인하세요.  |
+------------------------------------------------------------------------+
```

#### 구성 요소 및 동작
| 요소 | 설명 / 동작 |
|---|---|
| 진입(QR/링크/ID) | `verify/{tokenId}` 경로. QR 스캔·링크·홈 입력 모두 같은 화면으로 연결. |
| 검증 결과 카드 | `verify(tokenId)` 읽기 결과. 발급기관·활성여부·보유자·발급일·취소여부 표시. **지갑·트랜잭션 불필요.** |
| 상태 색상 | ✅유효(초록) / ⚠️기관 비활성(주황) / ❌취소·불일치(빨강). |
| [기관 프로필 ▸] | 발급기관 공개 프로필(UI-006)로 이동. |
| **[파일 업로드]** | 선택. 업로드 파일의 keccak256를 브라우저에서 계산 → 온체인 dataHash와 비교(파일은 어디에도 전송 안 됨). |
| 대조 결과 | 같으면 "원본 일치", 다르면 "해시 불일치 — 위변조 의심" + 두 해시 비교 표시. |
| [Etherscan ↗] | 발급 트랜잭션/토큰을 외부에서 독립 검증. |

#### 비고
- 발급 사실(✅)과 파일 무결성(원본 일치)은 **별개**다. 발급은 진짜인데 제출 파일만 위조된 경우 = "✅ 유효 + ❌ 해시 불일치"로 표시되어 *위조를 정확히 잡아낸다*. (시그니처 데모 포인트)

## 6. 테스트 시나리오

### 6.1 발급기관 등록 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 관리자 등록 | 관리자 지갑으로 발급기관 등록 | 등록 성공, 목록 표시 |
| 비관리자 등록 | 비관리자 지갑으로 등록 시도 | 등록 실패 |
| 중복 등록 | 이미 등록된 주소 재등록 | 거부 |
| 비활성/재활성 | 관리자가 발급기관 비활성→재활성 | 상태 전환 반영 |

### 6.2 권한 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 활성 발급기관 발급 | 등록 활성 기관으로 발급 | 발급 성공 |
| 미등록 주소 발급 | 미등록 지갑으로 발급 | 발급 실패 |
| 비활성 기관 발급 | 비활성 기관으로 발급 | 발급 실패 |
| 타 기관 취소 | 발급하지 않은 기관이 취소 시도 | 거부 |

### 6.3 발급 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 정상 발급 | 보유자·파일·종류 입력 후 발급 | 보유자 지갑에 SBT 민팅, 기록 저장 |
| 보유자 누락 | 보유자 주소 없이 발급 | 입력 검증 실패 |
| 해시 누락 | 파일 미선택 발급 | 입력 검증 실패 |

### 6.4 검증 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 유효 증명서 검증 | 유효 tokenId 조회 | 발급기관·발급일·유효 표시 |
| 원본 일치 | 원본 파일 업로드 | "원본 일치" 표시 |
| 위변조 파일 | 변조 파일 업로드 | "해시 불일치 — 위변조 의심" |
| 발급기관 비활성 | 비활성 기관 발급 건 검증 | 유효하되 "발급기관 비활성" 경고 |

### 6.5 취소(Revocation) 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 정상 취소 | 발급 기관이 취소 | 검증 시 "취소됨" 표시 |
| 중복 취소 | 이미 취소된 건 재취소 | 거부 |

### 6.6 소울바운드 테스트
| 케이스 | 절차 | 기대 결과 |
|---|---|---|
| 전송 시도 | 보유자가 증명서 transfer | revert(양도 불가) |

## 7. 릴리즈 기준

- 관리자가 발급기관을 등록/비활성할 수 있다.
- 등록된 활성 발급기관만 증명서를 발급할 수 있고, 미등록/비활성은 차단된다.
- 발급된 증명서는 보유자 지갑에 소울바운드로 민팅되며 양도가 차단된다.
- 검증자는 지갑 없이 QR/링크로 유효/취소/발급기관 활성 여부를 확인할 수 있다.
- 원본 파일 업로드 시 해시 일치/불일치가 정확히 표시된다.
- 발급 기관(또는 관리자)이 증명서를 취소할 수 있고, 검증에 반영된다.
- 온체인에 원본 파일/개인 식별 상세가 저장되지 않는다(해시·메타만).
- 주요 오류 상황 사용자 메시지가 제공된다.
- 라이브 URL 접속 가능, 데모 백업 영상 보유, 콘솔 에러 0, 반응형/다크라이트 정상.
- README + Sepolia 컨트랙트 주소(가능 시 verify) 공개.

## 11. 부록 (Appendix)

### 11.1 검증 가능한 자격증명 / 소울바운드 토큰 배경
- W3C **Verifiable Credentials**, **Open Badges**, **Blockcerts**(MIT 디지털 학위), Vitalik의 **Soulbound Token(SBT)** 개념이 본 프로젝트의 이론적 배경이다.
- 핵심 아이디어: 자격은 *발급자 서명*으로 증명되고, 보유자는 자신의 자격을 *소유*하며, 검증은 *탈중앙적으로* 가능해야 한다. Proofolio는 이 개념을 학습용 범위로 구현한다.

### 11.2 발급·검증 시퀀스 다이어그램
```text
시간 ─────────────────────────────────────────────────────▶

[등록]  Admin ── registerIssuer(addr,name) ──▶ Proofolio  (Issuer Active)

[발급]  Issuer  (오프체인) 원본 파일 → keccak256 = dataHash
        Issuer ── issueCredential(holder,dataHash,type) ──▶ Proofolio
                                       SBT 민팅(holder), 기록 저장
                                       ── CredentialIssued ──
[보관]  Holder  내 증명서 목록에서 확인, 이력서에 verify/{id} QR 삽입

[검증]  Verifier ── (지갑X) verify(tokenId) 조회 ──▶ Proofolio(read)
                  결과: 발급기관/활성/발급일/취소여부
        Verifier  원본 파일 업로드 → keccak256 비교 → ✅/❌

[취소]  Issuer/Admin ── revokeCredential(tokenId) ──▶ Proofolio
                  revoked=true ── CredentialRevoked ──
```

### 11.3 보안 감사 체크리스트
| # | 위협 | 대응 |
|---|---|---|
| S-1 | 권한 우회(비관리자 등록/비발급기관 발급) | Ownable2Step + issuer 권한 modifier |
| S-2 | 무단/남용 취소 | 발급한 기관만 취소. 관리자는 직접 취소 불가(기관 비활성으로 대응) → 중앙 권한 남용 차단 |
| S-3 | 증명서 거래/양도 | 소울바운드 `_update` 오버라이드: mint(from==0)만 허용, 보유자 간 전송 revert. ⚠️ mint 경로 막지 않도록 주의 |
| S-4 | 파일 위변조 | 원본 keccak256 해시 온체인 기록 + 검증 시 대조(정확한 원본 바이트 기준) |
| S-5 | 발급 해시 신뢰 한계 | dataHash는 발급기관 제출값(발급 시 정직성 가정). 컨트랙트가 직접 해싱하지 않음 — 문서화 |
| S-6 | 개인정보 영구 공개 | credType/metaURI에 PII 금지, 해시·일반 종류만. 보유 이력은 공개·열거 가능함을 고지 |
| S-7 | 발급기관 이름 사칭 | 신뢰 기준은 주소+관리자 등록(이름 아님). 검증 UI에 주소 항상 표시 |
| S-8 | 발급기관 키 탈취 | 관리자가 해당 기관 즉시 비활성화 + 영향 증명서 취소 |
| S-9 | 재진입/그리핑 | `_safeMint` 대신 `_mint` 사용(holder 컨트랙트 콜백 방지). 자금 이동 없음(저위험), CEI 준수 |
| S-10 | 가스 폭증/DoS | 상태 변경 함수에 무제한 루프 금지. `holder=>tokenIds` 매핑 + 프론트 이벤트 인덱싱 |
| S-11 | 입력 검증 | 주소!=0, 해시!=0, 이름/종류 비어있지 않음, 중복 등록·중복 취소 차단, 미존재 tokenId 차단 |
| S-12 | 정수 오버/언더플로 | Solidity ^0.8 기본 checked |
| S-13 | owner 키 분실/오전송 | `Ownable2Step`(2단계 수락) |
| S-14 | tokenURI/메타 신뢰 | 외부 metaURI 렌더 시 sanitize, 가능하면 온체인 최소 메타 |
| S-15 | 시크릿 노출 | PRIVATE_KEY는 .env(프론트 노출 금지), VITE_* 공개값만 |
| S-16 | 잘못된 검증 신뢰 | "발급기관 비활성/취소/해시 불일치"를 검증 UI에 명확히 경고 |

→ 각 항목은 Phase 3 테스트에서 케이스로 검증한다.

### 11.4 바이브코딩 노하우 수집 계획 (평가 20점 대비)
> "AI로 어떻게 만들었나"는 발표 20점 항목인데 많이들 놓친다. 빌드하면서 자동으로 쌓이도록 규율을 정한다.
- **단계별 커밋 규율**: Phase마다 `feat: pN ...` 커밋 + 1줄 회고. 커밋 히스토리 = 워크플로 증거.
- **프롬프트 로그**: 효과 좋았던 프롬프트 2~3개 기록/스샷(예: "PRD 전체를 컨텍스트로 제공", "소울바운드/권한 패턴 명시 요청").
- **AI 실수 사례 1개**: AI가 틀린 부분 + 발견·수정 과정(테스트로 잡음 등). 발표 하이라이트.
- **AI를 튜터로**: SBT·검증 가능 자격증명 개념을 AI로 학습·검증한 과정 기록.
- **검증 루틴**: AI 코드 → Hardhat 테스트 → 테스트넷 확인 → 코드 리딩(맹신 금지).
- **산출물**: `docs/vibe-coding-notes.md` + 발표 슬라이드 "나만의 바이브코딩 노하우" 1~2장.
